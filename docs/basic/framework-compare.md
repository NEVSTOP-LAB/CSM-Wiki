# CSM框架与其他框架对比

---

## 为什么需要编程框架

在 LabVIEW 项目开发中，工程团队通常会面临以下共性问题：

- **代码模块重复开发**，人力资源浪费
- **不同项目基于不同的开发框架**，维护学习成本高
- **人员梯队技能熟练度不同**，难以共同参与需协同的复杂项目开发
- **ATE自动化测试平台基于 OOP 编程**，项目集成商/客户难以自主维护
- **ATE标准自动化测试脚本化需求**无法满足

一个好的编程框架能系统性地解决上述问题，提高团队开发效率和代码质量。

## CSM 独特之处

CSM（可通讯状态机，Communicable State Machine）具有以下独特特点：

1. **纯文本的流程控制** — 消息、状态、参数全以字符串表示，可直接读写，天然支持外部脚本化控制
2. **隐形的"通讯总线"** — 模块间通过虚拟总线通讯，无需显式依赖队列或用户事件句柄
3. **VI 即模块** — 一个 VI 就是一个模块，利用 LabVIEW 原生调试手段（断点、探针、高亮执行）即可调试
4. **内置超级详细的检查接口** — 内嵌全局日志和调试工具，适合远程分析程序运行状态
5. **拓展式的设计** — 通过 Add-on 机制可按需扩展，不影响核心框架

## 主流框架对比

以下为 CSM 与 LabVIEW 社区常见框架的对比。

### 框架简介

| 框架 | 全称/说明 | 典型特点 |
| ---- | --------- | -------- |
| **DQMH** | Delacor Queued Message Handler | 基于队列消息处理，需要工具脚本生成 API，OOP 风格不明显 |
| **SMO** | JKI State Machine Objects | 基于 JKISM + LabVIEW OOP，Singleton/Standalone 模式 |
| **Actor Framework (AF)** | LabVIEW Actor Framework | LabVIEW 官方框架，高度依赖 OOP 和继承 |
| **CSM** | Communicable State Machine | 基于 JKISM 扩展，字符串消息通讯，非必须 OOP |

### 详细对比

| 对比维度 | DQMH | SMO | Actor Framework | CSM |
| -------- | ---- | --- | --------------- | --- |
| **外部接口** | Script 辅助创建 API | 类公共接口作为 API | 类封装的 Message 消息通讯 | 字符串格式的消息通讯，非必须创建 API |
| **状态反馈** | User Event，外部需要存在事件结构添加订阅逻辑 | User Event，外部需要存在事件结构添加订阅逻辑 | 消息处理中封装反馈消息 | 消息通过 JKISM 机制传递，无需 User Event Structure；隐式传递，订阅无需修改侵入代码，可以外部调用 |
| **可复制模块** | 复杂，两套模板 | 容易，类实现，框架不区分 | 容易，类实现，消息和模块独立，灵活性高 | 容易，VI 属性决定 |
| **代码依赖** | 调用方需要依赖模块的自定义事件和参数定义，需要预先开发框架定义部分的接口代码 | 调用方高度依赖模块实现，所以必须先开发框架接口部分 | 调用方可能有继承依赖和消息依赖，需预先定义消息类和方法 | 调用方开发无需依赖框架代码。传递消息、参数都使用字符串，不显式依赖，不容易 broken 代码。复杂参数需要依赖 |
| **参数传递** | 任意类型，使用 User Event 传递 | 任意类型，使用 User Event 传递 | 任意类型，需要封装 Message 类 | 字符串，复杂类型需转换（支持 MassData、API String 等扩展） |
| **代码复杂度** | 框架逻辑简单，模块代码冗余度高，不依赖 script 增删比较复杂 | 框架特别复杂，模块逻辑基于 JKISM，但开发需要理解 JKI SMO 的概念逻辑 | 框架特别复杂，实现高度依赖 OOP | 继承 JKISM 的优点，代码逻辑集中，理解消息通讯状态跳转后，与 JKISM 编程思路相同 |
| **生态** | 最佳（有完善工具和文档） | 一般，通常作为企业内部统一架构 | OOP（LabVIEW 原生） | 待完善（有社区支持，持续更新） |
| **执行效率** | 依赖 Event Structure | 依赖 Event Structure / LabVIEW OOP | LabVIEW OOP | 依赖参数的解析 |
| **UI 编写** | 支持 | 支持 | 不推荐 | 支持 |
| **RTOS 支持** | 支持 | 不推荐 | 支持 | 支持 |
| **高级模式** | 自定义模板 | 可组合，模块能作为子模块 | 继承 Actor Core.vi 实现特殊的模式 | 工作者模式 / 责任链模式 / 自定义模板 |

## CSM 与 QMH/DQMH 的消息机制对比

**QMH 的限制**：QMH 只有单向的消息通讯，无法轻松实现返回值机制。

**DQMH 的限制**：DQMH 需要使用工具添加 API，且 API 的类型（同步/异步）在添加时需要固定，后续修改需要重新生成。

**CSM 的消息机制优势**：

- 同一个消息可以被当作**同步、异步、异步无返回**消息使用，不需要预先固定类型
- 所有的消息都可以**携带返回**
- 所有的消息都**携带了执行时发生的错误**

```csm
// 同一个消息，三种调用方式
API: DoSomething >> Arguments -@ TargetModule   // 同步调用
API: DoSomething >> Arguments -> TargetModule   // 异步调用（有返回）
API: DoSomething >> Arguments ->| TargetModule  // 异步调用（无返回）
```

## CSM 与 Actor Framework 的订阅机制对比

**Actor Framework 的状态反馈**：通过 User Event 传递，外部模块需要在事件结构中添加订阅逻辑，与模块代码耦合。

**CSM 的订阅机制优势**：

- 通过**文本语句**控制订阅关系，而非用户事件句柄
- 订阅关系既可以由**模块内部**创建，也可以在**外部**创建
- 可**动态**创建和取消
- 模块**不存在**时也可以预先创建订阅
- 消息本身也能被订阅

```csm
// 建立订阅关系（不需要修改任何模块代码）
Data Sampled@DAQ >> API: Logging@Logging -><register>

// 取消订阅
Data Sampled@DAQ >> API: Logging@Logging -><unregister>
```

## 高内聚的模块设计

相比其他框架，CSM 追求更高内聚的模块设计：

- 模块关注于**内部功能实现**、**外部接口**（API）、**状态变化**（Status）
- 底层功能模块**不直接调用**其他模块接口（API、Queue、Event），可以通过发送状态广播去触发其他模块
- 可以通过**纯文本描述**模块的接口定义
- 可以预制**测试脚本**的方式定义模块行为
- **订阅关系可外部关联**，无需模块本身发起，实现真正的松耦合

### 与 DQMH 模块开发对比

| 开发场景 | DQMH | CSM |
| -------- | ---- | --- |
| 添加 API | 需要使用工具脚本生成，并事先确定 API 类型 | 直接在 Case 结构中添加，无需工具，同一 API 可同步/异步调用 |
| 模块测试 | 需要创建专用测试 VI | 直接用 CSM 调试面板发送消息测试，无需额外代码 |
| 模块文档 | 需要维护单独文档 | 可用纯文本描述接口文档，直接在调试面板中运行脚本验证 |
| 添加新模块 | 可能需要修改现有模块代码 | 通过订阅关系挂接，不修改现有代码 |

## 选择建议

| 场景 | 推荐框架 | 原因 |
| ---- | -------- | ---- |
| 企业级 LabVIEW 应用，团队经验丰富 | DQMH 或 CSM | 都有较好的工程实践支持 |
| 需要严格 OOP 设计的大型系统 | Actor Framework | OOP 继承机制更适合复杂层次结构 |
| 混合新老工程师团队 | CSM | 学习曲线平滑，新手可快速上手 |
| ATE 自动化测试脚本化 | CSM | 纯文本消息天然支持脚本化控制 |
| 需要模块插件化替换 | CSM | 虚拟总线设计，添加模块无需修改现有代码 |
| 偏远部署、需要远程调试 | CSM | 内嵌全局日志系统，适合运维场景 |

{: .note }
> 框架选择没有绝对的好坏，应根据团队实际情况、项目需求和已有代码库来决定。CSM 与 DQMH、SMO、Actor Framework 在实际工程中也可以互补使用。
