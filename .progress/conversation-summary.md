# CSM-Wiki 文档完善项目 - 对话总结

## 项目背景

这是一个针对CSM-Wiki文档的系统化完善项目，由用户（架构设计专家）发起，目标是根据参考资料补充和完善整个Wiki文档。

## 核心要求

### 用户原始需求（问题陈述）

```
你是一个架构设计的专家，现在正在更新这个文档。目前的文档已经完成了一部分，你要做的是根据参考文档对内容进行补充。
参考的资料在 .ref/ 目录，希望完成全部的文档内容在 readme.md 中有罗列。
```

### 详细要求（共9条）

1. **阅读理解**：首先阅读已有的文档内容和参考文档内容，理解目前要做的事情
2. **制定计划**：根据readme中待完成的项目，罗列详细的计划，覆盖readme中的全部任务
3. **逐步完成**：根据计划一步步完成所有的文档编写
4. **中文优先**：首先完成中文的编写，不考虑英文的版本
5. **使用辅助文件**：可以创建本地文件辅助记忆，放在统一指定的位置
6. **灵活组织**：可以创建新文档，也可以分割现有文档，只要更加合理
7. **质量保证**：每次完成一项任务时，都要自己审查本次修改，提出修改建议，至少三次反复修改，保证不遗漏内容。要记录：①修改；②参考内容；③审查意见及应对
8. **内容追踪**：.ref中vi description的所有内容在wiki中都要体现，创建简易跟踪文件保证没有遗漏
9. **专业写作**：行文专业、简洁流畅，可以创建流程图辅助说明，但不要滥用，也不要滥用表格、列表等

### 后续补充要求

- **项目记忆**：进度相关的临时文件要保存到仓库中，用于之后的"你"了解进展和情况，这就是你的"记忆"
- **持续迭代**：多次对话中用户都要求"继续"，表示对工作满意并希望继续推进

## 工作方法论

### 项目组织方式

1. **分阶段执行**：将整个项目分为5个阶段，每个阶段聚焦特定主题
2. **进度跟踪**：创建.progress/目录存储项目记忆文件
3. **参考资料追踪**：创建vi-description-tracking.md追踪参考资料覆盖情况
4. **频繁提交**：使用report_progress工具频繁提交进度，保持工作可追溯

### 文档完善流程

每个文档的完善遵循以下流程：

1. **调研阶段**
   - 查看README TODO LIST中的要求
   - 阅读.ref/目录中的参考资料
   - 查看现有文档内容

2. **规划阶段**
   - 制定详细的内容补充计划
   - 确定需要新增的章节
   - 规划示例和最佳实践

3. **编写阶段**
   - 根据参考资料扩充内容
   - 添加应用场景和代码示例
   - 补充最佳实践和使用建议
   - 增加常见问题解答

4. **审查阶段**（至少3轮）
   - 第1轮：内容完整性审查
   - 第2轮：技术准确性审查
   - 第3轮：可读性和流畅性审查

5. **提交阶段**
   - 使用report_progress提交进度
   - 更新进度跟踪文件
   - 记录完成情况

### 质量保证措施

1. **参考资料覆盖**：创建vi-description-tracking.md追踪所有参考资料的覆盖情况
2. **三轮审查机制**：每个文档至少经过3轮审查和修改
3. **内容追踪表**：确保参考资料中的所有内容都被覆盖
4. **行文标准**：保持专业、简洁、流畅的行文风格
5. **适度使用图表**：使用流程图、对比表等辅助说明，但不滥用
